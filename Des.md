# Scheduling Module Documentation

This document provides an overview of the `Scheduling` module, its structure, and the purpose of its key components. This module is a C# implementation of a discrete-event simulator for solving Job-Shop Scheduling Problems (JSP), with a focus on validation against well-known benchmark instances.

## Folder Structure

The `Scheduling` module is organized into the following subfolders:

-   `Core/`: Contains the core logic for the discrete-event simulator. These scripts are pure C# and have no Unity dependencies.
-   `Data/`: Contains data structures for representing scheduling problem instances, specifically the Taillard benchmark format.
-   `Unity/`: Contains `MonoBehaviour` components that bridge the core simulator with the Unity environment for in-editor validation and testing.
-   `Validation/`: Contains scripts and tools for validating the correctness and performance of the simulator. This includes comparing results against a reference Python implementation.

---

## `Core/`

This folder contains the heart of the scheduling simulator.

-   **`DESSimulator.cs`**: The main simulator class. It manages the simulation clock, events, jobs, and machines. It can load a problem instance and run the simulation from start to finish.
-   **`DispatchingRules.cs`**: Implements a variety of common dispatching (or priority) rules used in scheduling to decide which job to process next when a machine becomes available. Examples include Shortest Processing Time (SPT) and First-Come, First-Served (FCFS).
-   **`EventQueue.cs`**: A min-heap priority queue for managing simulation events. Events are ordered by their timestamp, ensuring the simulation proceeds in the correct chronological order.
-   **`Job.cs`**: Defines the `Job` and `Operation` classes. A `Job` consists of a sequence of `Operation`s, where each operation must be processed on a specific machine for a certain duration.
-   **`Machine.cs`**: Represents a machine on the shop floor. It can be idle or busy processing an operation and maintains a queue of jobs waiting to be processed.

---

## `Data/`

This folder handles the data representation of scheduling problems.

-   **`TaillardInstance.cs`**: A data structure designed to deserialize job-shop problem instances from JSON files that follow the Taillard benchmark format. It stores the processing times, machine sequences, and metadata (like the known optimal makespan).

---

## `Unity/`

This folder contains Unity-specific scripts that act as a bridge to the core simulation engine, allowing it to be used within the Unity Editor.

-   **`TaillardValidator.cs`**: A `MonoBehaviour` that you can attach to a GameObject. It provides a user interface in the Inspector to select a Taillard instance (from `Resources/Instances`), choose a dispatching rule, and run a validation test directly in the editor.
-   **`CrossValidator.cs`**: A `MonoBehaviour` used to automate the process of running the C# simulator over a batch of instances and exporting the results (makespans and full schedules) to CSV and JSON files. These exported files are formatted to be directly comparable with the output from a companion Python implementation.

---

## `Validation/`

This folder contains tools for ensuring the simulator is correct and for comparing its performance against a reference implementation.

-   **`ValidationRunner.cs`**: A class responsible for running a series of validation checks. It can run a single instance with a specific rule, or run all available rules to see which one performs best. It also includes methods to verify that fundamental scheduling constraints (like operation precedence and machine capacity) are not violated.
-   **`ValidationExporter.cs`**: This static class contains the logic to export simulation results into formats (CSV and JSON) that match those generated by the Python validation scripts. This enables a direct, line-by-line comparison of outputs to check for discrepancies.
-   **Python Scripts (`*.py`)**: This collection of Python scripts serves as an independent, reference implementation of a job-shop simulator.
    -   `generate_reference.py`: Generates the canonical `reference_makespans.csv` and `reference_schedules.json` files using the `job-shop-lib` Python library.
    -   `compare_results.py`: Compares the `csharp_makespans.csv` generated by `ValidationExporter` against the `reference_makespans.csv` and reports any differences in the calculated makespans.
    -   `compare_mirror.py`: Provides an even more detailed comparison by checking the full schedule JSON files for any discrepancies in start/end times for any operation.
    -   `mirror_des.py`: A minimal, standalone Python implementation of the same DES logic, used by `compare_mirror.py` to ensure the comparison is fair.
